# 设计模式速查表

> 源码中反复出现的 13 种设计模式，按使用频率排列

## 概述

Claude Code 源码中有一些反复出现的设计模式，它们不是教科书上的 GoF 模式，而是在 LLM agent 工程实践中自然演化出来的。本文汇总这些模式，便于快速检索和复用。

## 模式总览

| # | 模式名称 | 核心源文件 | 一句话描述 |
|---|---------|-----------|-----------|
| 1 | 决策冻结 | `toolResultStorage.ts` | 一旦决定是否替换某个工具结果，永远不再改变 |
| 2 | 草稿纸模式 | `compact/prompt.ts` | 让模型先在 `<analysis>` 中思考，再在 `<summary>` 中输出，最后剥离 analysis |
| 3 | 落盘+预览 | `toolResultStorage.ts` | 大结果写磁盘，上下文只放 2KB 预览 + 文件路径 |
| 4 | 逐字引用 | `compact/prompt.ts` | 要求模型逐字引用原文，防止多次压缩后意图漂移 |
| 5 | 熔断器 | `autoCompact.ts` | 连续失败 N 次后停止重试，防止无限循环 |
| 6 | 分层防御 | 多个文件 | 从源头到兜底的多层防线，成本递增、频率递减 |

## 模式详解

### 1. 决策冻结

**源文件**: `src/utils/toolResultStorage.ts`

**描述**: 每个 tool_result 的命运（是否被替换为磁盘引用）在第一次被看到时冻结。之后每轮都用缓存的替换内容重新应用，保证字节级一致。

**何时用**: 当下游系统（如 prompt cache）依赖内容稳定性时。一旦做出决定就不再改变，避免缓存失效。

**何时不用**: 当内容确实需要动态更新时（如实时数据）。冻结会导致过时数据无法被替换。

### 2. 草稿纸模式

**源文件**: `src/services/compact/prompt.ts`

**描述**: 压缩时要求模型先在 `<analysis>` 标签中分析，再在 `<summary>` 标签中输出摘要。`<analysis>` 部分被 `formatCompactSummary()` 剥离，不进入最终上下文。

**何时用**: 需要模型深度思考但不想让思考过程占用上下文时。类似 chain-of-thought 但不保留中间步骤。

**何时不用**: 当思考过程本身就是有价值的输出时（如代码审查的推理过程）。

### 3. 落盘+预览

**源文件**: `src/utils/toolResultStorage.ts`

**描述**: 工具结果超过 50K 字符时，完整内容写入磁盘文件，上下文中只放 2KB 预览和文件路径。模型可以用 Read 工具按需回查。

**何时用**: 工具可能产生大量输出（搜索结果、日志、大文件）。保证上下文不被单个结果撑爆。

**何时不用**: 当结果必须完整呈现给模型时（如需要全文分析的场景）。

### 4. 逐字引用

**源文件**: `src/services/compact/prompt.ts`

**描述**: 压缩摘要中要求模型逐字引用用户的原始任务描述，而不是用自己的话重新表述。防止多次压缩后任务描述逐渐偏离原意（"传话游戏"效应）。

**何时用**: 任何需要跨多轮保持信息准确性的场景。特别是会被反复摘要/压缩的关键信息。

**何时不用**: 当原文本身就很长，逐字引用会占用过多空间时。

### 5. 熔断器

**源文件**: `src/services/compact/autoCompact.ts`

**描述**: 连续自动压缩失败 3 次后停止重试。源码注释记录了曾有 1,279 个会话出现 50+ 次连续失败，每天浪费约 25 万次 API 调用。

**何时用**: 任何可能失败的自动重试逻辑。特别是失败后重试可能继续失败的场景（如上下文已经无法压缩）。

**何时不用**: 当每次重试都有合理的成功预期时（如网络抖动导致的临时失败）。

### 6. 分层防御

**源文件**: 五层防爆体系涉及多个文件

**描述**: 从 L1（源头截断）到 L5（兜底截断），每层解决不同粒度的问题。便宜的操作先做，昂贵的操作只在必要时触发。

**何时用**: 需要防御多种规模问题的系统。核心是"大多数情况用便宜的方案解决，只有极端情况才用昂贵方案"。

**何时不用**: 问题规模单一、不需要多级处理的简单场景。

### 7. 静态/动态分区

**源文件**: `src/constants/prompts.ts`

**描述**: System prompt 在 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 处分为两部分。静态区（身份、工具、规范）跨会话可缓存，动态区（语言、MCP 指令、环境信息）每会话不同。

**何时用**: 使用 prompt cache 的任何应用。把不变的内容放前面，变化的内容放后面。

**何时不用**: 不使用 prompt cache 或 prompt 很短的场景。

### 8. 恢复消息注入

**源文件**: `src/query.ts`

**描述**: 模型输出被 `max_output_tokens` 截断时，注入一条用户消息："Resume directly — no apology, no recap. Break remaining work into smaller pieces." 让模型无缝接续。

**何时用**: 模型输出可能被截断的场景。注入的消息要明确告诉模型"不要道歉、不要重复、直接继续"。

**何时不用**: 输出不会被截断的场景，或截断后不需要继续的场景。

### 9. 空结果标记

**源文件**: `src/tools/FileReadTool/`

**描述**: 文件内容未变时返回固定的 stub 字符串（"File unchanged since last read..."），而不是重新读取完整内容。通过 `readFileState` 的内容哈希追踪。

**何时用**: 同一资源可能被多次请求但内容不变的场景。用固定标记替代重复内容。

**何时不用**: 资源内容频繁变化，或每次都需要最新内容的场景。

### 10. memoize+手动清缓存

**源文件**: `src/services/mcp/client.ts`

**描述**: `connectToServer` 用 lodash `memoize` 缓存连接结果，相同配置不重复连接。但在认证失效（401）时手动清除缓存条目，触发重新连接。

**何时用**: 资源创建昂贵但偶尔需要刷新的场景。memoize 提供默认缓存，手动清除处理异常。

**何时不用**: 资源创建便宜或需要频繁刷新的场景。

### 11. 外部监督者

**源文件**: `src/utils/hooks.ts`

**描述**: Stop hooks 在模型宣布"完成"时执行外部检查。如果检查不通过，注入 blocking error 消息，强制模型继续工作。模型自己不知道有这个监督者。

**何时用**: 需要验证模型输出质量的场景。特别是模型可能过早宣布"完成"的长任务。

**何时不用**: 简单的问答场景，或模型输出不需要验证的场景。

### 12. 上下文继承

**源文件**: `src/tools/AgentTool/runAgent.ts`

**描述**: 子 agent 可以 fork 父 agent 的消息历史和 `readFileState` 缓存，共享 prompt cache 前缀。但有独立的 agentId 和 transcript。

**何时用**: 子任务需要父任务的上下文但不应该污染父上下文的场景。fork 而非共享。

**何时不用**: 子任务完全独立，不需要父上下文的场景（如并行的独立搜索）。

### 13. 流式并行执行

**源文件**: `src/StreamingToolExecutor.ts`

**描述**: 不等模型输出完毕，每个 tool_use block 完成后立即开始执行。多个工具调用可以并行执行。

**何时用**: 模型可能在一次输出中调用多个独立工具的场景。显著减少等待时间。

**何时不用**: 工具调用之间有依赖关系，必须串行执行的场景。

## 补充模式

以下模式在源码中同样反复出现，与上述 13 种模式互补。

### 14. 统一接口

**源文件**: `src/Tool.ts`

**描述**: 40+ 个工具都遵循同一个 `Tool<Input, Output, Progress>` 接口，通过 `buildTool()` 工厂函数构建。调用者不需要知道具体是哪个工具——所有工具都通过 `tool.call(input)` 执行。

**何时用**: 需要插件/工具系统的任何框架。统一接口让新工具的添加不需要修改调用方代码。

**何时不用**: 工具之间差异极大，强制统一接口会导致过度抽象。

### 15. 策略模式

**源文件**: `src/utils/permissions/PermissionMode.ts`

**描述**: 不同的权限模式（default/acceptEdits/bypassPermissions/plan）是不同的"策略"。运行时根据用户选择切换权限检查算法，而不是用 if/else 硬编码。

**何时用**: 同一个操作有多种处理方式，需要在运行时切换。

**何时不用**: 只有一种处理方式，策略模式会增加不必要的抽象层。

### 16. 渐进增强

**源文件**: `src/main.tsx`

**描述**: 基础工具（Bash、FileRead、FileEdit）总是可用，高级功能（Voice、Browser、LSP）通过 feature flag 按需启用。未启用的功能甚至不会被加载到内存（`require()` 而非 `import`）。

**何时用**: 功能集合庞大但大部分用户只需要子集的场景。保证基础功能稳定，高级功能不影响启动速度。

**何时不用**: 所有功能都是核心功能，不存在可选子集。

### 模式的组合

实际系统中模式很少单独出现。以工具执行流程为例：

```
统一接口 → 所有工具用同一个 call() 方法
  + 策略模式 → 不同的权限检查策略
  + 事件驱动 → Hook 在执行前后触发
  + 分层防御 → 多层安全检查
  + 缓存策略 → 工具 schema 缓存
  = 一个既灵活又安全又高效的工具执行系统
```
